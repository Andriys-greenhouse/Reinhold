2. Scenarios

  1. Use cases
    User can use application in following ways:

    • Personal assistant
      If user for some reason prefers Reinhold to other free personal assistants he can use his limited abilities in this field to do things like dial in for a phone call, search web for keyword, get random number and similar actions.

    • Chatting companion
      Reindold have some pre-programmed answers and neural network to decide by which of them to respond, user can experiment with inputs and explore Reinholds capabilities.

    • Interesting technological toy
      User can use Reinhold as toy to explore limits and benefits of my implementation of this technology.

    • TTS
      User can use Reinhold for reading short texts.

    • Complete usage
      Using all of the functionalities mentioned above.

  2. Personas - types of user roles
    This application won't have pre-implemented personas, no user will be restricted in using the application in any way.

    Users of this application might be numerous, students trying to get the concept of conversational AI or of Android apps, teachers demonstrating something on the open-source code, tech enthusiasts who want to explore this application, users, who want to be sure, that conversational AI which they are playing with, is not sending any data about them outside their devices, and many others.

  3. Delimitation of the scope

    • Functionality modules
      Application won't be able to complete any order (concerning performing some action) given by user except pre-defined orders. Order to complete this operation will be recognized by Reinhold in more than one form (it shouldn't be only a defined command but number of actions which could be performed will be limited by defining them in code - Reinhold won't be able to complete any command which won't be pre-defined).
      These include:

      • perform a phone call - Reinhold will dial in a specified phone number or contact

      • play word football - simple game where (after user will “say” the first word) players will have to alternately say words which start at the same word with which last word ended (every word can be used only once), when one of players won't be able to think of any more word which match conditions, he looses

      • play rock paper scissors - after user will input his answer Reinhold will compare it with a randomly drawn answer of his

      • play guess a number from 1 to 10 - compared with randomly drawn number

      • pick random number from … to … - returns a random number from the range specified by the user

      • book quote - Reinhold will return user's favorite quote from this book or will search for some on the internet

      • random story - return random story

      • battery percentage left

      • current time

      • info about Reinhold - pre-defined answer on some questions concerning Reinhold himself

      • search on web - Reinhold will write out few results from the google web sear API

      • news - returns some actual news articles

      • read out loud - Reinhold will use his TTS to read users input

    • TTS
      TTS will be implemented using Xamarin.Essentials.

    • Data saving
      Data needed for operation of the app (information about user, his acquaintances, his books, his stories, information about when were the app started and so on) are stored as serialized Json strings in SecureStorage (part of Xamarin.Essentials) and Properties of application. Some data which is expected to be larger (texts of stories, quotes,…) are going to be stored separatly.
      Safety of the data shall be ensured by use of SecureStorage from Xamarin.Essentials as means to store this data.

    • Use of third party content
      For the development of this app, third-party content will be used. Separate document concerning used resources will be written and submitted along with final product. For example icons are going to be drawn from the web. Similarly, other used resources and software are going to be mentioned as well.

    • Reinhold's responses
      Responses of the Reinhold shall be pre-defined. Sometimes there will be randomly selected from possible answers but Reinhold won't “make up hi own answers”.

    • Color scheme
      There are going to be three color schemes red, green, and blue. Color scheme will effect color of most icons, color of message “bubbles” and of headers.

    • Accessibility
      This software won't be published on Google Play or similar big stores but potential users will be able to download complete project from GitHub and compile it themselves.

    • Feedback
      No feedback feature of the application shall be implemented. Users will have to submit their feedback onto a GitHub.

  4. Areas with low emphasis

    • Login
      This app won't include any method of confirming the user's identity. User will have to realize his own security precautions to prevent abuse of the app by unauthorised person (for example install app which enables him to secure launching of specified app by applying necessity to input password).

    • Backup
      No backup system will be implemented. Once user will delete or overwrite his data, there will be no way of getting it back. User will be notified about this before he will confirm any big modification of data.
      Data will be saved every time when user will leave the application.

    • Responsibility
      Author bears no responsibility for any modification or lose of data to which user will give app access to (calendar, contacts,…).

    • Performance
      No more than necessary amount of emphasis will be put on applications smooth run (app will be implemented with basic requirements of async programming but minimal requirements won't be a priority).

    • Game intelligence
      Reinhold will play pre-defined games with no AI involved. (defined behavior with randomness involved)

    • Help
      No help shall be build in. If user will want to look up something concerning this app, he can do so in this document.


3. Rough overall architecture

  1. Workflow
    Described in attached Diagram.

  2. Pages and their details
    Graphic design is shown in Diagram.

    • Flyout menu
      From the Flyout menu user can access Chat page, Data management page and Settings page. Flyout menu will be accessed by pressing icon depicting three lines on the top left of every of main pages.

      • Main components:

        Header - displays information about version of the application and about the version of the core.

        Bottom screen menu - Although it isn't part of the menu itself, it will enable the user to switch between pages in an easier way.

    • Chat page
      In Chat page, user will be able to interact with Reinhold by writing messages as keyboard inputs.

      • Main components:

        Messages - Every message will contain information about date and time when it were added.

        Input text field - Place where user will be able to write his messages to.

        Microphone/send button - If there is text in the Input text field this button will submit the message to the app.

    • Settings page
      This page will enable the user to set some basic values.

      • Main components:

        Reset button - This button will reset all settings to default and it will delete all of user's data. User will be warned about function of this button after pressing it and he will have to confirm his choice.

      • Settable values:

        Color scheme - User can pick from tree colors (more about this in Delimitation of the scope).

        Display notifications - Either the user will get notification about Reinhold's new Automatic question (this feature might be added in future versions) or not.

    • Data management page
      From this page user will be able to get to other pages to edit data stored about him.

      • Main components:

        Manage existing data buttons - Four buttons enabling the user to go to the Create/edit person page (with users data), List of stories page, List of books page and List of acquaintances page.

        Add data buttons - This way user can be redirected to empty Create/edit story page, Create/edit book page or Create/edit person page.

    • List of books page & List of stories page & List of acquaintance page
      This page will contain list of registered objects with possibility to add new ones.

      • Main components:

        Existing XXX elements - In list view every book will be represented by its title, acquaintance by it's full name and story by few of its initial words. When element is clicked, user will be redirected to Edit/create book page, Edit/create person page or to Edit/create story page where user can edit element from the line on which he clicked on. Bin button on the right will enable user to delete record about element in the line.

        Add XXX - User will be redirected to Edit/create book page, Edit/create person page or to Edit/create story page with no data filled in (if he will fill some data in, after clicking add button new object will be added to the memory).

    • Edit/create book page & Edit/create person page & Edit/create story page
      In this page user will be able to modify data about existing object or to add data about new ones.

      • Main components:

        Done button - Firstly validity of entered contacts (at editing person) is going to be checked (if the contact will be invalid a notice about what is wrong shall appear). Than if contacts are going to pass validation, new object can be created.

  3. Core description

    Simple description
    The core composes of two neural networks. These have been trained by inputs which were supposed to simulate user-given input and they are stored in a .json file in Assets of application.
    One of networks is responsible for recognizing context in which given input is directing a conversation and the other one should recognize “what the user wants” (intent).
    Input of networks consists of current context and of bag of words of input.
    After core “analyzes” input, required data is extracted from user-given input to serve as an input for methods (dates, phone numbers and similar) to complete the order.
